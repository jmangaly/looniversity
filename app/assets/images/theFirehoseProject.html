Lesson 17

Testing user login & adding login links to the navbar

In the last step we installed devise, which adds all the user authentication to our application. Now lets go and test our new user authentication.

So if we go back and run this command in the terminal

$ rake routes

We'll see the new pages that got setup by devise. There are a couple of particularly interesting routes that got setup that we should check out:

new_user_session      GET  /users/sign_in(.:format)      devise/sessions#new
new_user_registration GET  /users/sign_up(.:format)      devise/registrations#new
new_user_password     GET  /users/password/new(.:format) devise/passwords#new

First let's just look at the sign-up page. Go to this page we see in our routes http://localhost:3030/users/sign_up. Take a moment and sign up for a new user account with your web application.

Nice that looks like it's the page that will allow users to sign up. Similarly the other urls that we pointed out will allow users to sign in, and reset their password.

So now that we know devise is installed, let's add some links to the relevant pages in the navbar.

If we keep looking in the devise documentation under the getting started section we can see the section for Controllers, filters and helpers. This is telling us two interesting ways that we'll be able to use devise. It says we can both add a before_filter :authenticate_user! to require that a user is authenticated before accessing a page, or we can access the current signed-in user using current_user. It's also telling us we can check to see if a user is signed in by running code that looks like user_signed_in?.

Ok, with those three things we can do a lot in making the user authentication work fully.

First in the right-hand part of the topnav let's add some code to display if a user is logged in.

Open app/views/layouts/application.html.erb and adjust the file to look like this:

<!-- Collect the nav links, forms, and other content for toggling -->
  <div class="collapse navbar-collapse" id="bs-example-navbar-collapse-1">
    <ul class="nav navbar-nav">
      <li>
        <%= link_to new_place_path do %>
          <i class="glyphicon glyphicon-plus-sign"></i>
          New Place
        <% end %>
      </li>

    </ul>
    <ul class="nav navbar-nav navbar-right">

      <% if user_signed_in? %>
        <li><%= link_to 'A user is signed in!', '#' %></li>
      <% else %>
        <li><%= link_to 'No user is signed in :(', '#' %></li>
      <% end %>
      <li><a href="#">Link</a></li>
    </ul>
  </div><!-- /.navbar-collapse -->
</div><!-- /.container-fluid -->

Cool, save the file and refresh. Nice, so if you created a user in the last step, you should already be signed in so it's showing you the message that you're signed in. Awesome :)

Now, instead of just saying 'A user is signed in!', let's make it be a link to sign the user out, and make it so when they click the link the user signs out.

Ok, so if we go back to the terminal and run this:

$ rake routes

We can see this line:

destroy_user_session DELETE /users/sign_out(.:format)      devise/sessions#destroy

This looks like it will allow us to sign out. So when we build the link we need to use the destroy_user_session_path url helper and also specify that we need to treat this as a DELETE action. Cool, let's apply this change in the topnav now.

<% if user_signed_in? %>

  <li><%= link_to 'Sign out', destroy_user_session_path, :method => :delete %></li>
<% else %>
  <li><%= link_to 'No user is signed in :(', '#' %></li>
<% end %>

Save the file and refresh the page. We see that the link has been updated to 'Sign out'. Now press that link and you should see the message that 'No user is signed in :('.

Perfect. Now let's switch the No user is signed in :( link to be a 'Sign in' link so users can sign in.

If we run:

$ rake routes

The line that we care most about now, which will allow users to sign in, is this line:

new_user_session GET    /users/sign_in(.:format)       devise/sessions#new

So we want to make a link to the new_user_session_path location (and since it's a GET request we can just leave the method to be the default).

Adjust the code and make that change inside app/views/layouts/application.html.erb:

<% if user_signed_in? %>
  <li><%= link_to 'Sign out', destroy_user_session_path, :method => :delete %></li>
<% else %>

  <li><%= link_to 'Sign in', new_user_session_path %></li>
<% end %>

Next, let's make it so if a user isn't signed in we will display a link to "Sign Up", in addition to "Sign In". If we look in our rake routes the line that looks interesting for this use case is this one:

new_user_registration GET  /users/sign_up(.:format)  devise/registrations#new

It looks like we'll be using the new_user_registration_path for the link.

Let's make that change now.

<% if user_signed_in? %>
  <li><%= link_to 'Sign out', destroy_user_session_path, :method => :delete %></li>
<% else %>
  <li><%= link_to 'Sign in', new_user_session_path %></li>

  <li><%= link_to 'Sign up', new_user_registration_path %></li>
<% end %>

Save the file and refresh. Our topnav should be working as we intend. The registration pages don't look too beautiful right now. In the next lesson we're going to make it so we actually start using users in the back-end and then in the following lesson we'll go ahead and clean up those pages and make them look awesome.

How to get a fast answer for your question:

Here’s how to ask smart questions in the forum:


First, give it a good title that describes what you’re struggling with. Descriptive headlines like “my quotes don’t show on my index page“ or “the email notification email is not triggered at comment submission” work best.


Second, describe in detail what you’ve done up to this point. For example, you should say things like: “I added a quote through the rails console and can see it in my database. I also added the code to create the loop on my index.html.erb page and added the necessary code inside of my quotes controller too”.


Third, describe in detail what you expected to happen. For example, you should describe things like this: I refreshed my browser window with the url “localhost:3030” and expected a new quote to be shown on my root page.


Fourth, paste a screenshot of your error message. This one is obvious, if we can see what you’re seeing we can help you a lot better. Make sure you get the whole page.


Fifth, include a link to your github repository: If we can’t figure out how to help you through your description above, we often times go directly into GitHub and take a look at your web application code, pull it onto our machines, and give it a little test drive.

Important: Make sure you always commit and push the latest version of your code to your GitHub repository, before submitting your question in the Q&A forum. By doing so we can look at your latest code and see the same error message as you.


Sixth, include a link to you web application: We might need to have a look at your app live on the internet. Including the url for your app will help. Plus if you don't include the url, you might get a response from Jarvis, the firehose robot, requesting that you add it.


Speed of help: We generally check out forum several times each day and answer all questions. The better you formulate your problem, the faster you get your solution.

This makes sense. If you forget to include a screenshot of your error message, the most likely reply that you get back will be something like “hey there, can you please include a screenshot of your error message so we can see what you’re seeing?”.


Directly emailing your Mentor a question: The forum is a lot faster way to get help - since we always have somebody on dedicated forum watch. It might seem tempting to go around the forum and email your mentor directly, but that will most likely get you a delayed response, since your mentor might be helping other students and will not get back to you as fast.
